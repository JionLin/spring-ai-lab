# spring-ai-lab 长期演进计划（以本仓库为唯一基底）

本文档约定：**不另起仓库**，在现有分层（`web` / `tools` / `external` / `domain`）上增量演进。验收以「可运行 + 可测试 + 文档可复述」为准。详细能力对照见 `docs/项目能力与Spring-AI工程化对照说明.md`。

---

## 一、目标与原则

### 1.1 总目标

在 6～7 个月尺度内，把本仓库从「最小 Chat + Tool + Redis 向量」演进为可同时演示：

1. **对话产品化能力**：流式、多轮会话、可选结构化输出。  
2. **RAG 工程能力**：文本 → Embedding → 入库/更新 → 检索增强生成（Advisor 或等价编排）。  
3. **企业化底线**：可观测、关键路径韧性、认证与工具治理。

### 1.2 两条路线（按目标选择）

| 路线 | 适用场景 | 覆盖阶段 | 预计周期 |
|------|----------|----------|----------|
| **面试优先** | 3～4 个月内需要面试 | 0 → 1A/2 并行 → 1B → 3 → **M1** → 5（薄）→ 6（薄）→ **M2** | **12～16 周** |
| **完整工程** | 不赶时间，做扎实 | 0 → 1 → 2 → 3 → 4 → 5 → 6 → 7 | **24～28 周** |

> 面试优先路线中：1A = 流式（薄版）、1B = 会话记忆、M1/M2 = 面试里程碑。阶段 4 与阶段 7 在面试路线中降级为可选。

### 1.3 架构原则（写代码时约束）

| 原则 | 做法 |
|------|------|
| 控制器瘦身 | `ChatController` 只做 HTTP 适配；编排进 `application` 或 `domain.chat` 包 |
| 可替换模型 | 继续沿用 profile（`qwen` / `deepseek`），Embedding 与 Chat 密钥/端点分配置 |
| 向量双轨过渡 | 短期保留 `VectorSearchService`；新增写入路径与索引 schema 变更需有迁移说明 |
| 测试不绑真模型 | 集成测试使用 Mock、Testcontainers 或录制响应；`mvn verify` 在 CI 可过 |

### 1.4 与 OpenSpec 的关系（可选）

每个大阶段可对应 `openspec/changes/<name>/`：先 `proposal.md` + `tasks.md`，再按 `openspec-apply-change` 技能拆 PR，避免一次性巨型合并。

## 二、现状基线（不做重复设计）

- **已有**：`ChatClient` + `@Tool`、`ExternalJsonPostClient`（重试/熔断）、`VectorSearchService`（RediSearch KNN）、用户查询、traceId、多 profile LLM。  
- **缺口**：流式、会话记忆、Embedding 与文档摄入、Spring AI Advisor 式 RAG 编排、结构化输出、LLM 专用指标、JWT/工具鉴权等（见对照文档第八章）。

## 三、阶段总览（依赖关系）

```text
阶段 0 仓库与配置基线
    ├──→ 阶段 1 对话工程底座（流式 + 会话）
    └──→ 阶段 2 RAG 数据面（Embedding + 文档 + 写入向量）  ← 与阶段 1 无依赖，可并行
              ↓
         阶段 3 RAG 控制面（检索编排 / Advisor + 与 Chat 集成）
              ↓
         ★ 里程碑 M1：面试可讲 RAG 全链路
              ↓

              ↓
         阶段 6 安全与多租户会话
              ↓
         ★ 里程碑 M2：完整叙事可演示
              ↓
         阶段 7（可选）混合检索 / Rerank / 评测

  ──── 横切 ────
  可观测（贯穿阶段 1～6，每阶段验收时补一条 metric）
```

**关键变化**：阶段 1 与阶段 2 **无技术依赖**，可并行启动或交替推进；可观测不再占独立阶段编号，改为横切关注点。

---

### 阶段 1：对话工程底座——流式 + 会话（约 3～4 周）

| 项 | 内容 |
|----|------|
| 交付物 | 新端点 `POST /chat/stream`（`text/event-stream`）；`conversationId` 维度多轮；历史持久化（Redis，复用现有 `spring.data.redis`） |
| 前端 Demo | `src/main/resources/static/index.html`：一个 `EventSource` + 简易对话框，**不引入前端框架**；用于演示与录屏 |
| 包建议 | `domain.chat`：`ChatSessionStore`、`ChatOrchestrator`；`web` 仅委托 |
| 技术要点 | `ChatClient.stream()`、客户端断开时取消订阅、UTF-8 SSE；`Message` 列表上下限（条数或 Token 粗估截断） |
| 可观测（横切） | 为 `/chat` 和 `/chat/stream` 增加耗时 metric（Micrometer Timer） |
| 验收 | 浏览器打开 `index.html` 能看到流式 chunk；同 `conversationId` 第二轮能引用前文；metric 端点可查 |

**依赖**：阶段 0。

> **面试优先路线**：可先做流式薄版（1A，约 1 周，仅 SSE 端点 + 静态页），立即并行启动阶段 2；会话记忆（1B）在阶段 2 期间或之后补。

---

### 阶段 2：RAG 数据面——Embedding + 文档摄入 + 向量写入（约 2～4 周）

| 项 | 内容 |
|----|------|
| 交付物 | 支持至少 Markdown（可选 PDF）；Chunk 策略可配置；调用 Embedding 模型生成向量；写入 RediSearch 索引（与 `VectorSearchService` 共用字段约定） |
| 依赖 `pom` | 按 Spring AI 1.0.x 官方文档增加 Embedding 所需依赖；PDF 则增加解析库（注意许可证） |
| 包建议 | `domain.rag.ingestion`：`DocumentLoader`、`Chunker`、`EmbeddingPort`；`infra.embedding`：DashScope/OpenAI 兼容实现 |
| 配置 | 新增 `application-rag.yml` 或在现有 yml 增加 `lab.rag.*`：chunkSize、overlap、索引名、批大小 |
| 可观测（横切） | Embedding API 调用耗时、文档摄入条数 |
| 验收 | 给定样例文件，运行一次摄入后 `FT.SEARCH` 能命中；单元测试覆盖 chunk 边界与空文档 |

**依赖**：阶段 0；Redis 向量 profile 可用。**与阶段 1 无依赖，可并行**。

> 时间从原计划 4～8 周调为 **2～4 周**：你已有 Redis KNN 基础设施与索引约定，新增核心工作为「文档读取 → 切分 → 调 Embedding API → 写入」，不含从零搭建向量库。

---

### 阶段 3：RAG 控制面——检索编排 + 与对话集成（约 3～5 周）

| 项 | 内容 |
|----|------|
| 交付物 | 用户问句 → Embedding → `VectorSearchService`（或封装接口）→ 片段排序/截断 → 注入 `UserMessage`/`SystemMessage`；**优先**使用 Spring AI **Advisor** 能力（以当前 BOM 版本文档为准）；若 API 受限则用同一语义的服务类 + 单测 |
| 包建议 | `domain.rag.retrieval`：`RetrievalService`（TopK、score 阈值、Token 预算）；与 `ChatOrchestrator` 组合 |
| 与 Tool 关系 | 保留 `WeatherTools`；新增「知识库问答」路径：或新接口 `/chat/rag`，或由编排层根据意图选择「仅 RAG / RAG+Tool」（阶段 7 可强化） |
| 可观测（横切） | 检索耗时、命中条数、注入 context 长度 |
| 验收 | 文档中有答案的问题能引用检索片段；日志可看到检索耗时与命中条数 |

**依赖**：阶段 2 完成数据写入；阶段 1 建议已完成（多轮追问更贴近面试场景，但非硬依赖）。

## 五、可观测——横切关注点（贯穿阶段 1～6）

不再作为独立阶段。每阶段验收时必须同步完成对应 metric：

| 阶段 | 新增 Metric |
|------|-------------|
| 1 | `/chat` 与 `/chat/stream` 请求耗时（Timer）；`ai_estimated_cost_total`（Counter，按 usage tokens × 单价累加） |
| 2 | Embedding API 调用耗时、文档摄入条数（Counter）；Embedding 调用纳入 `ai_estimated_cost_total` |
| 3 | 检索耗时、命中条数、注入 context token 估算 |
| 6 | 鉴权失败次数、工具调用审计（Counter） |

**`ai_estimated_cost_total` 实现要点**：

- 数据来源：`ChatResponse.getMetadata().getUsage()`（`promptTokens` + `completionTokens`）；Embedding 同理。
- 单价配置：`lab.cost.price-per-1k-tokens.chat` / `lab.cost.price-per-1k-tokens.embedding`，按 profile 区分（qwen 与 deepseek 单价不同）。
- 流式注意：`stream()` 模式下 token 计数通常只在**最后一个 chunk** 返回，需在流结束回调中统一采集，不要每个 chunk 都累加。
- 标签建议：`model`（区分 qwen/deepseek）、`call_type`（chat/embedding）、`tool_invoked`（是否触发工具）。
- 面试价值：面试官问「成本怎么控制」时可直接指向 Prometheus 指标，而非口头说「看日志」。

**基础设施**：引入 `spring-boot-starter-actuator` + Micrometer；可选 Prometheus / OTel 导出。

## 八、风险与应对

| 风险 | 应对 |
|------|------|
| DashScope/DeepSeek API 与 Spring AI 版本差异 | 锁定 BOM；大版本升级单独开分支与回归清单 |
| **Spring AI 快速迭代**（1.0.x → 下一大版本可能 break Advisor/Memory API） | 6 个月计划内大概率遇到；升级前先在分支验证，保留旧版兼容分支 |
| RediSearch 索引与 Java 侧维度不一致 | 写入与检索共用常量与集成测试 |
| Controller 膨胀 | Code Review 拒绝大段业务进 `web` 包 |
| 真模型费用 | 本地开发 profile 使用 mock / 小模型 / 限流 |
| **Embedding/Chunking 学习曲线**（首次做 RAG 的 Java 工程师） | 预留 1 周纯学习+小实验时间在阶段 2 开头；先用官方示例跑通再接入本项目 |
